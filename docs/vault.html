Vault & Encryption

AES-256-GCM local encryption for secrets. Nothing leaves your machine.

Overview

Gate Vault provides local-only AES-256-GCM encryption for sensitive values. Encrypt database URLs, API keys, and other secrets so they never exist as plaintext in your repository or deployment configuration. All encryption and decryption happens on your machine — secrets never leave your infrastructure.

Local Only

All encryption happens on your machine. No external services, no network calls, no cloud dependencies.

secure

AES-256-GCM

Industry-standard authenticated encryption. Random IV per operation, authentication tag prevents tampering.

cryptography

Auto-Decryption

Environment variables with the VAULT: prefix are automatically decrypted at server startup.

convenience

Key Management

Generate a vault key with a single command:

gate vault keygen

This creates ~/.gate/vault.key containing 32 random bytes encoded as hex (64 hex characters). The file is automatically set to chmod 0600 (owner read/write only).

Never commit your vault key. The key file lives at ~/.gate/vault.key outside your repository. Add it to your global gitignore if you want extra safety. Back it up securely — if you lose the key, encrypted values cannot be recovered.

Encrypting Values

Encrypt a single value from the command line:

gate vault encrypt "postgresql://user:password@host:5432/mydb"

Output is a base64-encoded JSON blob containing:

FieldDescription
algoAlgorithm identifier (aes-256-gcm)
ivRandom initialization vector (base64)
ctCiphertext (base64)
authTagGCM authentication tag (base64)

A new random IV is generated for every encryption operation, so encrypting the same value twice produces different output.

Decrypting Values

Decrypt a vault blob back to plaintext:

gate vault decrypt "eyJhbGdvIjoiYWVzLTI1Ni1nY20iLC..."

The vault key at ~/.gate/vault.key must be present. If the authentication tag does not match (tampered data), decryption fails with an error.

Encrypting .env Files

Encrypt an entire .env file in one step:

gate vault env .env

This creates .env.encrypted where every value is replaced with a VAULT:-prefixed encrypted blob:

# Original .env
DATABASE_URL=postgresql://user:password@localhost:5432/gate_dev
REDIS_URL=redis://localhost:6379
STRIPE_SECRET_KEY=sk_live_abc123...

# Generated .env.encrypted
DATABASE_URL=VAULT:eyJhbGdvIjoiYWVzLTI1Ni1nY20iLC...
REDIS_URL=VAULT:eyJhbGdvIjoiYWVzLTI1Ni1nY20iLC...
STRIPE_SECRET_KEY=VAULT:eyJhbGdvIjoiYWVzLTI1Ni1nY20iLC...

Comments and empty lines are preserved. The encrypted file is safe to commit to your repository.

VAULT: Prefix Auto-Decryption

Environment variables prefixed with VAULT: are automatically decrypted when the Gate server starts. This means you can use encrypted values directly in your .env or system environment:

# In your .env or system environment
DATABASE_URL=VAULT:eyJhbGdvIjoiYWVzLTI1Ni1nY20iLC...

# Gate reads this at startup and decrypts it to:
DATABASE_URL=postgresql://user:password@localhost:5432/gate_dev

The decryption is transparent — application code sees the plaintext value. The vault key must be available at ~/.gate/vault.key on the machine running the server.

Use Cases

Database Credentials

Encrypt DATABASE_URL and REDIS_URL so connection strings are never stored as plaintext in configuration files or CI environments.

Payment Keys

Encrypt STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET, and other payment provider credentials.

API Keys

Encrypt third-party API keys like GITHUB_CLIENT_SECRET, SLACK_WEBHOOK_URL, SENDGRID_API_KEY, and others.

CI/CD Secrets

Store encrypted .env.encrypted in your repo and decrypt at deploy time using the vault key available on the production server.

Security Properties

PropertyDetail
EncryptionAES-256-GCM (authenticated encryption with associated data)
IV generationCryptographically random, unique per encryption operation
IntegrityGCM authentication tag detects any tampering with ciphertext
Key storageLocal filesystem only (~/.gate/vault.key), chmod 0600
Key never committedKey lives outside the repository at ~/.gate/
No networkAll operations are fully offline, no external service calls
Forward secrecyRandom IV per operation means identical plaintext produces different ciphertext

Backup your vault key. If you lose ~/.gate/vault.key, all values encrypted with that key are permanently unrecoverable. Store a backup in a secure password manager or hardware security module.

Quick Reference

# Generate vault key (once)
gate vault keygen

# Encrypt a single value
gate vault encrypt "my-secret-value"

# Decrypt a vault blob
gate vault decrypt "eyJhbGdv..."

# Encrypt entire .env file
gate vault env .env

# Result: .env.encrypted with VAULT: prefixed values
# Use .env.encrypted as your env file — Gate auto-decrypts at startup
